## 当前方案：SQLite + JavaScript 计算

### 实现方式

系统使用 SQLite 存储向量，通过 JavaScript 进行相似度计算：

```javascript
// 存储方式
embedding TEXT  // JSON 字符串存储 1024 维向量

// 相似度计算
cosineSimilarity(vec1, vec2) {
  // JavaScript 循环计算余弦相似度
  // 时间复杂度: O(n)，n = 向量维度
}

// 向量搜索
async vectorSearch(query, topK) {
  // 1. 生成查询向量
  const queryVector = await this.embeddingService.getEmbedding(query);

  // 2. 获取所有向量
  const allChunks = await this.getAllChunks();

  // 3. 逐一计算相似度
  const scoredChunks = allChunks.map(chunk => {
    const similarity = this.cosineSimilarity(queryVector, chunk.embedding);
    return { ...chunk, score: similarity };
  });

  // 4. 排序并返回 topK
  return scoredChunks.sort((a, b) => b.score - a.score).slice(0, topK);
}
```

## SQLite 方案分析

### ✅ 优点

1. **简单易用**
   - 无需额外服务
   - 零配置成本
   - 单文件存储

2. **开发友好**
   - 调试方便
   - 易于备份
   - 迁移简单

3. **资源占用低**
   - 内存占用小
   - CPU 开销可控
   - 磁盘 I/O 少

4. **成本为零**
   - 完全免费
   - 无云服务费用
   - 无需 API key

### ❌ 缺点

1. **性能瓶颈**
   - 全表扫描: O(n)，n = 知识块数量
   - 线性搜索: 每个向量都要计算
   - 响应时间: 随数据量增长

2. **内存消耗**
   - 所有向量加载到内存
   - 大数据集会占用大量 RAM

3. **无索引优化**
   - 无向量索引
   - 无法利用数据库优化

### 性能测试预估

| 知识块数量 | 查询时间 | 内存占用 | 适用场景 |
|----------|---------|---------|----------|
| 1,000 | < 50ms | ~4MB | ✅ 完全够用 |
| 10,000 | 200-500ms | ~40MB | ⚠️ 可接受 |
| 50,000 | 1-2s | ~200MB | ❌ 偏慢 |
| 100,000 | 2-5s | ~400MB | ❌ 不可接受 |
| 1,000,000 | 20-50s | ~4GB | ❌ 完全不可用 |

- ✅ 迁移风险低

## 性能优化建议

### 短期优化（当前架构）

```javascript
// 1. 添加查询缓存
class VectorKnowledgeBase {
  constructor() {
    this.cache = new Map(); // 或使用 Redis
  }

  async vectorSearch(query, topK) {
    const cacheKey = `${query}-${topK}`;
    if (this.cache.has(cacheKey)) {
      return this.cache.get(cacheKey);
    }

    const results = await this.search(query, topK);
    this.cache.set(cacheKey, results, 3600); // 缓存1小时
    return results;
  }
}

// 2. 使用 Worker 线程
const { Worker } = require('worker_threads');

async parallelSimilarity(queryVector, chunks) {
  const workers = [];
  const chunkSize = Math.ceil(chunks.length / 4); // 4个Worker

  for (let i = 0; i < 4; i++) {
    const worker = new Worker('./similarity-worker.js', {
      workerData: chunks.slice(i * chunkSize, (i + 1) * chunkSize)
    });
    workers.push(worker);
  }

  const results = await Promise.all(workers.map(w => w));
  return results.flat();
}

// 3. 预加载向量
class VectorKnowledgeBase {
  async initialize() {
    this.allChunks = await this.getAllChunks();
    // 向量已在内存中
  }

  async vectorSearch(query) {
    const queryVector = await this.embeddingService.getEmbedding(query);

    // 直接使用内存中的向量
    return this.allChunks
      .map(chunk => ({
        ...chunk,
        score: this.cosineSimilarity(queryVector, chunk.embedding)
      }))
      .sort((a, b) => b.score - a.score)
      .slice(0, 5);
  }
}
```

## 结论

### 对于当前项目

**SQLite 完全足够！**

**理由**:
1. 当前数据规模（< 1,000 知识块）远未达到瓶颈
2. 查询响应时间 < 100ms，用户体验良好
3. 零成本、零配置，开发效率高
4. 迁移成本远大于收益

### 何时需要迁移

**迁移触发条件**（满足任一即考虑）:
- ✅ 知识块数量 > 50,000
- ✅ 平均查询时间 > 1s
- ✅ 内存占用 > 2GB
- ✅ 用户投诉性能问题

**推荐迁移时机**:
- 知识块数量达到 20,000 时开始评估
- 达到 50,000 时执行迁移

### 最终建议

```
现在 → 继续使用 SQLite ✅

6个月后 → 根据数据规模决定 🤔

如需迁移 → 优先选择 Qdrant 🎯
```
